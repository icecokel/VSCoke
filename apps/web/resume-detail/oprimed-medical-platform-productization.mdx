## 개요 (Overview)

의료·임상 데이터 분석 플랫폼에서 시뮬레이션 입력, 서버 데이터 연동, 결과 리포트, 인증·세션, 프로젝트 실행 환경과 배포 준비 흐름을 정리했습니다.

내부 프로젝트 약어나 제품명을 몰라도 읽히도록 의료·임상 분석, 환자 조건, 약물 반응, 결과 예측, 리포트처럼 이해 가능한 표현을 중심으로 정리합니다.

## 1. 의료·임상 분석 입력 흐름 구조화

### 문제

의료·임상 분석 화면은 환자 기본 정보, 질병, 약물, 검사값, 분석 조건, 예측 결과가 서로 연결되어 있어 단순 입력 폼처럼 나누면 사용 흐름이 쉽게 끊길 수 있었습니다.

### 판단

화면을 기능 단위로만 나누지 않고, 사용자가 분석 조건을 입력하고 실행한 뒤 결과를 확인하는 순서와 서버에 전달해야 하는 데이터 구조를 함께 기준으로 잡았습니다.

### 실행

- 약물 반응 예측, 환자 하위그룹 식별, 임상 결과 예측 등 여러 분석 흐름의 입력, 검증, 실행 흐름을 개발했습니다.
- 결과 테이블, 차트 확대 보기, 문서 미리보기와 다운로드, 엑셀 파일 불러오기 흐름을 연결했습니다.
- 사용자가 분석 조건 입력부터 결과 확인까지 이어갈 수 있도록 화면 상태와 서버 요청 데이터를 맞췄습니다.

### 결과

사용자가 복잡한 의료·임상 분석 조건을 입력하고, 시뮬레이션 실행 후 결과 확인과 문서·엑셀 출력까지 이어갈 수 있는 프론트엔드 흐름을 구성했습니다.

## 2. 서버 데이터 명세와 화면 상태 정합화

### 문제

서버에서 요구하는 데이터 형식이 자주 바뀌고, 값이 비어 있는 경우나 질병별 입력 조건이 달라 화면 입력값과 실제 요청 데이터가 어긋날 가능성이 있었습니다.

### 판단

화면에서 임의로 값을 보완하는 방식을 줄이고, 서버 기능 명세 문서를 기준으로 요청·응답 타입, 서비스 함수, 화면 상태, 실행 조건을 맞추는 방식으로 정리했습니다.

### 실행

- 환자, 질병, 약물, 분석 실행, 결과 조회에 필요한 서버 요청·응답 타입을 정리했습니다.
- 성별·인종 코드, 약물 식별값, 분석 반복 조건을 서버 기능 명세 기준으로 보정했습니다.
- 저장된 입력값과 엑셀 파일 데이터를 다시 불러오는 예외 상황을 보정해 화면 상태와 서버 요청 데이터 사이의 불일치 가능성을 줄였습니다.

### 결과

서버 명세 변경에 대응하기 쉬운 구조로 정리하고, 화면 입력값과 실제 서버 요청값 사이의 불일치 가능성을 줄였습니다.

## 3. 결과 리포트와 공유 흐름 강화

### 문제

분석 결과가 단순 데이터 출력에 머물면 연구자가 결과를 해석하거나 공유하기 어렵고, 제품 화면으로서 완성도가 떨어질 수 있었습니다.

### 판단

결과 확인을 테이블, 차트, 문서 출력, 엑셀 파일 활용, 리포트 배치가 이어지는 하나의 사용 흐름으로 보고 화면과 출력 기능을 함께 정리했습니다.

### 실행

- 결과 테이블과 결과 요약 문구를 정리했습니다.
- 차트 확대 보기, 문서 미리보기와 다운로드, 엑셀 파일 불러오기와 내보내기 흐름을 구성했습니다.
- 디자인 시안 기준 레이아웃과 타이포그래피를 보정했습니다.

### 결과

분석 실행 이후 결과를 화면에서 확인하고 문서나 파일 형태로 활용할 수 있는 제품 경험을 강화했습니다.

## 4. 제품군 인증·세션 경험 통일

### 문제

여러 프론트엔드 제품마다 로그인, 세션 만료, 제품 이동, 모바일 접근 정책이 흩어져 사용자 경험과 유지보수 기준이 달라질 수 있었습니다.

### 판단

개별 제품별 예외 처리를 늘리기보다 공통 진입점과 공통 세션 만료 경험을 기준으로 제품군 인증 흐름을 통일했습니다.

### 실행

- 로그인 진입 화면, 인증 쿠키, 세션 복원, 제품 이동 라우팅 흐름을 정리했습니다.
- 세션 만료 공통 모달과 인증 실패 경험을 연결했습니다.
- 이전 접속 경로 기반 로그인 이동, 모바일 안내와 접근 제한 흐름을 정리했습니다.

### 결과

제품 간 이동과 인증 실패 경험을 일관되게 만들고, 여러 프론트엔드 제품이 같은 인증 흐름 위에서 동작할 수 있게 보강했습니다.

## 5. 프로젝트 실행 환경·배포 기준 정합화

### 문제

여러 프론트엔드 프로젝트의 Next.js와 React 버전, 의존성 잠금 파일, 컨테이너 실행 환경의 Node.js 버전, 로컬 보안 접속 환경, 자동 검증, 단독 실행 방식의 배포 준비 흐름이 서로 달라 운영과 유지보수 리스크가 커질 수 있었습니다.

### 판단

버전 맞춤을 단순 의존성 업데이트가 아니라 앱 실행 환경, 컨테이너 실행 환경, 의존성 잠금 파일, 배포 스크립트, 배포 전 검증 스크립트를 함께 맞추는 운영 안정성 문제로 봤습니다.

### 실행

- Next.js 업그레이드 검증을 문서화했습니다.
- React와 Next.js 보안 패치, Next.js 패치 버전 갱신, PostCSS 보안 패치 고정을 정리했습니다.
- 컨테이너 Node.js 버전 정합화, 공통 설정 반영, GitHub Actions와 Next.js 단독 실행 방식의 배포 스크립트를 정리했습니다.

### 결과

프로젝트별 실행 환경과 의존성 기준 차이를 줄이고, 개발-검증-패키징-배포 준비 흐름을 같은 기준으로 관리할 수 있게 보강했습니다.

## 성과 (Outcome)

복잡한 의료·임상 분석 워크플로우를 프론트엔드 제품 흐름으로 구조화하고, 서버 데이터 연동·결과 리포트·인증·세션·프로젝트 버전·배포 기준을 함께 정리해 여러 프론트엔드 제품을 운영 가능한 상태로 안정화했습니다.
