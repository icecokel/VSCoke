## 개요 (Overview)

**FE 운영·배포 흐름 정리**는 Nexus, Portal, Trial, Labs FE의 로컬 개발, 검증, 패키징, 배포 준비 흐름을 정리한 작업입니다.

여러 FE 제품이 각각 다른 배포 방식과 개발 환경을 갖고 있어, 산출물 전달과 검증 기준을 일관되게 맞추는 것이 중요했습니다. GitHub Actions, Docker, Next standalone, self-hosted runner, 로컬 HTTPS 환경을 중심으로 개발-검증-배포 흐름을 보강했습니다.

## 주요 구현 (Key Features)

### 1. CI/CD workflow 정리

- **GitHub Actions 구성**:
  - Nexus, Portal, Trial, Labs FE의 release/deploy/demo-deploy workflow와 배포 스크립트를 정리했습니다.
  - self-hosted runner 대상 설정, legacy container 처리, demo runner 배포 흐름을 정리했습니다.
- **검증 흐름 연결**:
  - typecheck, build, Playwright, Vitest, Husky pre-push/preparation 흐름을 정리했습니다.
  - 배포 전 확인해야 하는 검증 명령과 산출물 흐름을 문서와 스크립트 기준으로 맞췄습니다.

### 2. 패키징과 런타임 정리

- **Docker와 tar packaging**:
  - Dockerfile, tar packaging, load script를 추가하거나 정리해 FE 산출물을 전달 가능한 형태로 만들었습니다.
  - 배포 환경에서 필요한 정적 자산 경로와 환경 변수 흐름을 보정했습니다.
- **Next standalone runtime**:
  - Next standalone runtime 실행 흐름과 health check를 정리했습니다.
  - 제품별 nginx/env, static asset path, 실행 스크립트가 어긋나는 부분을 보정했습니다.

### 3. 로컬 개발 환경 보강

- **로컬 HTTPS 구성**:
  - mkcert 기반 로컬 HTTPS와 로컬 도메인을 구성해 실제 제품 도메인과 가까운 개발 환경을 만들었습니다.
  - 개발 env와 배포 env를 분리해 로컬 검증과 운영 배포 준비 흐름을 구분했습니다.
- **운영 문서화**:
  - runner sample/guide, 배포 스크립트, QA, design parity, progress 문서를 정리해 유지보수 가능한 기준을 남겼습니다.

## 기술적 포인트 (Technical Depth)

- **GitHub Actions**: 제품별 release/deploy/demo-deploy workflow를 정리하고 self-hosted runner 실행 조건을 맞췄습니다.
- **Docker/Next standalone**: FE 산출물 패키징과 standalone runtime 실행 흐름을 구성했습니다.
- **개발 환경**: `mkcert`, 로컬 도메인, env 설정을 통해 실제 제품 접근 환경에 가까운 검증 환경을 만들었습니다.

## 결과 (Outcome)

여러 FE 제품의 개발, 검증, 패키징, 배포 준비 흐름을 정리해 기능 구현 이후 운영 가능한 상태로 전달하고 검증할 수 있는 기준을 만들었습니다.
