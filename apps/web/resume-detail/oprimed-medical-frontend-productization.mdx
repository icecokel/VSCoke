## 기본 정보

- 회사: 오프리메드
- 기간: 2026.02 - 2026.06 (5개월)
- 역할: 프론트엔드 개발
- 핵심 포지션: 의료·임상 분석 프론트엔드 제품화 및 운영 기준 정리
- 기술 스택: Next.js, React, TypeScript, TanStack Query, Zustand, React Hook Form, ECharts, Plotly, Playwright, GitHub Actions, Docker, Shell Script

## 핵심 요약

의료·임상 분석 제품에서 환자 정보, 질병, 약물, 검사값, 분석 조건, 예측 결과가 복잡하게 연결되는 화면을 개발했다. 단순히 화면을 구현하는 데서 멈추지 않고, 여러 프론트엔드 프로젝트의 실행 환경, 보안 패치, 자동 검증, 배포 준비, 인증·세션, 성능 개선 기준을 함께 정리해 팀이 반복해서 겪던 유지보수 문제를 줄이는 방향으로 작업했다.

## 1. 프론트엔드 실행 환경, 보안 패치, 배포 검증 흐름 정리

### 문제

여러 프론트엔드 프로젝트의 Next.js와 React 버전, 의존성 잠금 파일, 컨테이너 실행 환경, 배포 방식, 로컬 보안 접속 방식이 서로 달랐다. 이 때문에 프로젝트마다 실행과 검증 기준을 다시 확인해야 했고, 배포 전 점검 흐름도 일관되게 유지하기 어려웠다.

### 접근

버전 업데이트를 단순 패키지 교체가 아니라 팀의 실행·검증·배포 기준을 맞추는 문제로 봤다. 애플리케이션 실행 환경, 컨테이너 실행 환경, 의존성 잠금 파일, 자동 검증, 배포 스크립트가 같은 방향으로 맞아야 유지보수 리스크를 줄일 수 있다고 판단했다.

### 실행

- Next.js, React, PostCSS 보안 패치와 의존성 잠금 파일을 정리했다.
- Docker 실행 환경의 Node.js 버전을 애플리케이션 실행 기준에 맞췄다.
- GitHub Actions 기반 자동 검증과 배포 준비 흐름을 정리했다.
- Docker 패키징, standalone 배포 스크립트, load/deploy shell script를 보강했다.
- self-hosted runner, health check, static asset path, 로컬 HTTPS 실행 기준을 정리했다.
- Playwright 기반 명령어형 브라우저 테스트와 실패 리포트 흐름을 추가했다.

### 결과

프로젝트별로 흩어져 있던 실행, 검증, 패키징, 배포 준비 기준을 정리해 팀의 유지보수 부담과 배포 전 확인 리스크를 줄였다. 새 환경에서도 같은 방식으로 실행하고 점검할 수 있는 기준을 마련했다.

## 2. 의료·임상 분석 입력 흐름과 서버 데이터 연동 정리

### 문제

의료·임상 분석 화면은 환자 정보, 질병, 약물, 검사값, 분석 조건, 예측 결과가 서로 연결되어 있었다. 입력값이 많고 조건이 복잡해 단순 입력 폼처럼 나누면 사용자가 흐름을 이해하기 어렵고, 화면 상태와 서버에 전달하는 요청 데이터가 어긋날 가능성이 있었다.

### 접근

화면을 기능 단위로만 보지 않고, 사용자가 분석 조건을 입력하고 실행한 뒤 결과를 확인하는 순서와 서버가 요구하는 데이터 구조를 함께 기준으로 삼았다. 화면 상태, 요청·응답 타입, 실행 조건, 저장된 입력값 복원 흐름을 같은 기준으로 맞추는 것이 중요하다고 판단했다.

### 실행

- 환자 정보, 질병, 약물, 검사값, 분석 조건 입력을 단계형 흐름으로 구조화했다.
- 서버 기능 명세 변경에 맞춰 요청·응답 타입과 서비스 함수를 정리했다.
- 성별, 인종, 약물 식별값, 분석 반복 조건 등 서버 요청 데이터의 변환 기준을 보정했다.
- 저장된 분석 조건과 엑셀 데이터를 다시 불러올 때 생기는 예외를 처리했다.
- 실행 조건과 필수 입력값 기준을 정리해 잘못된 요청 가능성을 줄였다.

### 결과

복잡한 의료·임상 분석 조건을 입력, 검증, 실행, 결과 확인까지 이어지는 제품 흐름으로 정리했다. 서버 기능 명세가 바뀌거나 저장값을 다시 불러오는 상황에서도 화면 입력값과 실제 요청 데이터의 불일치 가능성을 줄였다.

## 3. 결과 리포트, 차트, 문서·엑셀 활용 흐름 구성

### 문제

분석 결과가 단순 데이터 출력에 머물면 연구자나 운영자가 결과를 해석하고 공유하기 어렵다. 결과 화면은 표와 차트만 보여주는 것이 아니라, 분석 이후 문서로 확인하거나 엑셀 파일로 다시 활용하는 흐름까지 이어져야 했다.

### 접근

결과 확인을 하나의 화면 작업이 아니라 `결과 해석 -> 리포트 확인 -> 문서 미리보기와 다운로드 -> 엑셀 활용`으로 이어지는 업무 흐름으로 봤다. 차트와 리포트는 시각 요소가 아니라 사용자가 분석 결과를 판단하고 공유하기 위한 도구로 정리했다.

### 실행

- 분석 결과 테이블과 결과 요약 문구를 구성했다.
- ECharts, Plotly 기반 차트와 차트 확대 보기 흐름을 구현했다.
- 문서 미리보기와 다운로드 흐름을 정리했다.
- 엑셀 파일 불러오기와 내보내기 흐름을 분석 조건 복원과 결과 활용 관점에서 연결했다.
- 결과 리포트 배치, 모달, 날짜 선택, 빈 데이터 상태, 비활성 액션 같은 예외 상태를 보정했다.

### 결과

분석 실행 이후 결과를 화면에서 확인하는 데 그치지 않고, 문서와 엑셀 파일로 다시 활용할 수 있는 흐름을 만들었다. 결과 화면을 단순 표시 영역이 아니라 업무 판단과 공유에 쓰이는 제품 경험으로 구성했다.

## 4. 제품 인증 포털과 세션 경험 정리

### 문제

여러 프론트엔드 제품마다 로그인, 세션 만료, 제품 이동, 모바일 접근 정책이 달랐다. 제품별 예외가 늘어나면 사용자 경험이 달라지고, 인증 실패 상황을 유지보수하기 어려워질 수 있었다.

### 접근

인증을 각 제품의 개별 기능으로 보지 않고, 제품군 전체의 공통 진입점과 세션 경험으로 봤다. 로그인 이후 이동 흐름, 세션 만료 안내, 이전 접속 경로 복원, 모바일 접근 안내를 공통 기준으로 맞추는 방향으로 정리했다.

### 실행

- 여러 제품으로 이동하는 공통 로그인 화면과 제품 카드 화면을 개선했다.
- 인증 쿠키와 로그인 이후 이동 흐름을 보정했다.
- 세션 만료 시 공통 모달을 노출하도록 정리했다.
- 이전 접속 경로 기반 로그인 이동 흐름을 보정했다.
- 모바일 접근 안내와 접근 제한 흐름을 정리했다.

### 결과

제품별로 흩어져 있던 로그인, 세션 만료, 제품 이동 경험을 공통 기준으로 묶었다. 사용자 입장에서는 인증 실패 상황을 더 일관되게 이해할 수 있고, 팀 입장에서는 제품군 인증 흐름을 같은 기준으로 유지보수할 수 있게 보강했다.

## 5. 프론트엔드 성능과 공통 UI 구현 기준 개선

### 문제

일부 화면은 시각 효과를 정적 이미지 자산에 의존하고 있어 초기 렌더링과 정적 파일 요청 부담이 있었다. 또한 버튼, 아이콘, 토글, glass container 같은 UI 패턴이 화면마다 다르게 구현되면, 팀이 같은 UI를 반복해서 다시 판단해야 하는 비용이 커질 수 있었다.

### 접근

이미지를 단순히 압축하는 것보다, 이미지로 표현하던 UI를 코드 기반 공통 컴포넌트로 전환하는 것이 렌더링 부담과 유지보수 부담을 함께 줄일 수 있다고 판단했다. 성능 개선은 감으로 주장하지 않고, 정적 자산 제거량과 Lighthouse 결과처럼 설명 가능한 근거를 기준으로 정리했다.

### 실행

- 이미지 기반 glass frame을 CSS 기반 공통 컨테이너로 전환했다.
- runtime PNG 요청 최대 3건과 약 315 KiB payload를 제거했다.
- Lighthouse 비교 문서 기준 Performance score 88에서 98, Largest Contentful Paint 1.3s에서 1.0s, Speed Index 3.4s에서 0.7s 개선을 확인했다.
- 공통 버튼, 아이콘, 토글, glass container 계열을 정리해 UI 구현 방식의 편차를 줄였다.
- 미사용 정적 이미지, 미사용 코드, 미사용 의존성 정리 후보를 추적해 번들링과 유지보수 부담을 줄이는 방향으로 정리했다.

### 결과

이미지와 공통 UI 구현 방식을 코드 기반 기준으로 정리해 초기 렌더링 부담을 낮췄다. 동시에 팀이 같은 UI 패턴을 반복해서 다시 판단해야 하는 비용을 줄이고, 공통 컴포넌트 기준으로 화면을 확장할 수 있는 기반을 마련했다.

## 최종 성과

Oprimed 경력의 핵심은 단순 화면 구현이 아니라, 복잡한 의료·임상 분석 제품을 팀이 운영 가능한 프론트엔드 기준으로 정리한 경험이다. 화면 구현, 서버 데이터 연동, 결과 리포트, 인증·세션, 보안 패치, 자동 검증, 배포 준비, 성능 개선을 함께 다루며 제품과 팀의 반복 문제를 줄이는 방향으로 기여했다.
