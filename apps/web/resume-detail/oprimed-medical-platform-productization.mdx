## 개요 (Overview)

의료·임상 데이터 분석 플랫폼에서 복잡한 시뮬레이션 워크플로우, API 계약, 결과 리포트, 인증·세션, 프로젝트 버전·배포 흐름을 프론트엔드 관점에서 정리했습니다.

단순 기능 나열보다 문제를 어떻게 정의했고, 어떤 기준으로 화면 상태와 API 계약을 맞췄는지, 그 결과 어떤 제품 흐름을 만들었는지를 중심으로 정리합니다.

## 1. 의료·임상 시뮬레이션 워크플로우 구조화

### 문제

의료·임상 시뮬레이션 화면은 환자 정보, 질병, 약물, baseline feature, 분석 조건, 결과 데이터가 서로 얽혀 있어 단순 CRUD처럼 화면을 나누기 어려웠습니다.

### 사고/판단

화면 상태를 컴포넌트 편의 기준이 아니라 API payload와 사용자 진행 단계 기준으로 나누고, 입력-검증-실행-결과 확인 흐름이 끊기지 않게 구조화했습니다.

### 실행

- DRD, TRP, SUB, TOP, Twin Outcome 등 워크플로우의 입력, 검증, 실행 흐름을 개발했습니다.
- 결과 테이블, 차트 fullscreen, PDF preview/download, Excel import 흐름을 연결했습니다.
- 사용자가 분석 조건 입력부터 결과 확인까지 이어갈 수 있도록 화면 상태와 API 요청값을 맞췄습니다.

### 결과

사용자가 복잡한 의료·임상 분석 조건을 입력하고, 시뮬레이션 실행 후 결과 확인과 PDF/Excel 출력까지 이어갈 수 있는 프론트엔드 흐름을 구성했습니다.

## 2. API 계약과 화면 상태 정합화

### 문제

Swagger/OpenAPI 계약 변경과 nullable 값, 코드 매핑, 질병별 분기 때문에 화면 fallback이나 임시 mock이 남아 있으면 런타임 오류와 잘못된 요청값이 발생할 수 있었습니다.

### 사고/판단

임의 fallback을 줄이고, API 계약을 기준으로 타입, 서비스 레이어, 화면 상태, 실행 조건을 맞추는 contract-first 방식으로 정리했습니다.

### 실행

- 환자, 질병, 약물, 시뮬레이션, 결과 API 요청·응답 타입을 정리했습니다.
- 성별·인종 코드, 약물 id, bootstrap 조건을 API 계약 기준으로 보정했습니다.
- hydrate/import 복원 edge case를 보정해 화면 상태와 API 요청값 사이의 불일치 가능성을 줄였습니다.

### 결과

FE-BE 계약 변경에 대응하기 쉬운 구조로 정리하고, 화면 입력값과 실제 API 요청값 사이의 불일치 가능성을 줄였습니다.

## 3. 결과 리포트와 공유 흐름 강화

### 문제

분석 결과가 단순 데이터 출력에 머물면 연구자가 결과를 해석하거나 공유하기 어렵고, 제품 화면으로서 완성도가 떨어질 수 있었습니다.

### 사고/판단

결과 확인을 테이블, 차트, PDF, Excel, 리포트 배치가 이어지는 하나의 사용 흐름으로 보고 UI와 출력 기능을 함께 정리했습니다.

### 실행

- 결과 테이블과 overview text를 정리했습니다.
- 차트 fullscreen, PDF 미리보기/다운로드, Excel import/export 흐름을 구성했습니다.
- Figma 기준 레이아웃과 타이포그래피를 보정했습니다.

### 결과

분석 실행 이후 결과를 화면에서 확인하고 문서/파일 형태로 활용할 수 있는 제품 경험을 강화했습니다.

## 4. 제품군 인증·세션 경험 통일

### 문제

Portal, Labs, Nexus, Trial 등 제품군마다 로그인, 세션 만료, 제품 이동, 모바일 접근 정책이 흩어져 사용자 경험과 유지보수 기준이 달라질 수 있었습니다.

### 사고/판단

개별 제품별 예외 처리보다 Portal 중심 진입점과 공통 세션 만료 경험을 기준으로 제품군 인증 흐름을 통일했습니다.

### 실행

- Portal 로그인 게이트, 인증 쿠키, 세션 복원, 제품 라우팅 흐름을 정리했습니다.
- 세션 만료 CDN SDK와 공통 모달 흐름을 연결했습니다.
- referrer redirect, 모바일 안내/접근 차단 흐름을 정리했습니다.

### 결과

제품 간 이동과 인증 실패 경험을 일관되게 만들고, 여러 FE 제품이 같은 인증 흐름 위에서 동작할 수 있게 보강했습니다.

## 5. 프로젝트 버전·배포 기준 정합화

### 문제

여러 FE 프로젝트의 Next.js/React 버전, package lock, Docker node runtime, 로컬 HTTPS, CI, standalone 배포 흐름이 서로 달라 운영과 유지보수 리스크가 커질 수 있었습니다.

### 사고/판단

버전 맞춤을 단순 dependency bump가 아니라 `앱 런타임`, `컨테이너 런타임`, `lockfile`, `배포 스크립트`, `검증 스크립트`를 함께 맞추는 운영 안정성 문제로 봤습니다.

### 실행

- Next.js 업그레이드 검증을 문서화했습니다.
- React/Next 보안 패치, Next 패치 버전 갱신, PostCSS 보안 패치 고정을 정리했습니다.
- Docker node version alignment, fe-config 기반 설정 반영, GitHub Actions와 standalone 배포 스크립트를 정리했습니다.

### 결과

프로젝트별 런타임과 패키지 기준 차이를 줄이고, 개발-검증-패키징-배포 준비 흐름을 같은 기준으로 관리할 수 있게 보강했습니다.

## 성과 (Outcome)

복잡한 의료·임상 분석 워크플로우를 프론트엔드 제품 흐름으로 구조화하고, API 계약·결과 리포트·인증·세션·프로젝트 버전·배포 기준을 함께 정리해 여러 FE 제품을 운영 가능한 상태로 안정화했습니다.
