## 핵심 요약

React/Next.js/TypeScript 기반 의료·임상 분석 제품의 프론트엔드 개발을 담당했습니다.
복잡한 분석 화면의 서버 데이터 연동, UI 컴포넌트 마이그레이션, 렌더링 성능 개선을 중심으로 작업했고, 배포 전 검증과 CI/CD 흐름은 프론트엔드 개발 환경 안에서 함께 정비했습니다.

## 1. React 화면의 서버 데이터 연동과 사용 흐름 개선

### 문제

- 여러 화면에서 같은 서버 데이터를 따로 조회해 재요청과 캐시 기준이 흩어져 있었습니다.
- 분석 결과에서 다음 분석으로 이어질 때 사용자의 선택값과 진행 상태가 안정적으로 이어져야 했습니다.
- 일부 사용 흐름은 실제 동작과 화면 상태가 어긋나 오류, 빈값, 로딩 처리가 흔들릴 수 있었습니다.

### 해결·결과

- TanStack Query 기반 공통 조회 구조로 전환해, 질병 목록을 직접 호출하던 6곳을 제거했습니다.
- 분석 결과 화면에서 후속 분석 생성, 선택값 전달, 화면 전환을 연결해 다음 분석까지 이어지는 흐름을 구현했습니다.
- 동작하지 않거나 재정의가 필요한 구간을 분리해 공유하고, 오류·빈값·로딩 처리 기준을 개선하는 근거로 연결했습니다.

## 2. React UI 컴포넌트 마이그레이션과 렌더링 성능 개선

### 문제

- 일부 UI가 이미지에 의존해 불필요한 네트워크 요청이 발생했습니다.
- 화면마다 버튼, 아이콘, 색상 구현 방식이 달라 수정 범위가 커질 수 있었습니다.

### 해결·결과

- 이미지 기반 UI를 코드 기반 공통 컴포넌트로 전환해 PNG 요청 3건과 약 315 KiB 전송량을 제거했습니다.
- Lighthouse 비교 기준 LCP를 1.3s에서 1.0s로, Speed Index를 3.4s에서 0.7s로 낮춰 렌더링 속도 개선을 확인했습니다.
- 정적 이미지 WebP 변환 흐름을 추가해 이미지 리소스 전송량을 약 95% 줄였습니다.
- 공통 컴포넌트 사용 기준을 도입해 기존 버튼 import 25건과 정적 SVG 파일 34개를 제거했습니다.
- 직접 색상값 사용을 529건 줄여 색상 관리 기준을 단순화했습니다.

## 3. 프론트엔드 개발 환경과 CI/CD 검증 흐름 정비

### 문제

- React/Next.js 프로젝트마다 의존성, 실행 환경, 배포 전 확인 기준이 달라 빌드와 검증 과정에서 같은 문제를 반복 확인해야 했습니다.
- 커밋·푸시 단계에서 잡을 수 있는 타입 오류와 빌드 오류가 배포 준비 단계까지 넘어갈 수 있었습니다.
- GitHub Actions self-hosted runner를 팀이 같은 기준으로 세팅하고 사용할 수 있도록 참고 가이드가 필요했습니다.

### 해결·결과

- 보안 패치, 의존성 잠금 파일, 컨테이너 실행 환경, 프론트엔드 런타임 버전을 맞춰 React/Next.js 실행 환경을 표준화했습니다.
- GitHub Actions 기반 CI/CD와 Husky pre-commit/pre-push hook을 도입해 typecheck, build, Playwright 검증 흐름을 연결했습니다.
- Docker 패키징, Next.js standalone 실행, load/deploy 스크립트 기준을 정리하고 GitHub Actions self-hosted runner 세팅 가이드를 팀에 공유했습니다.
- Playwright CLI 기반 검증 체계를 도입해 주요 브라우저와 화면 크기 조합에서 반복 검증할 수 있는 기준 사례를 만들었습니다.
