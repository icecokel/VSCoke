## 기본 정보

- 회사: 주식회사 코드크레용
- 기간: 2024.09 - 2025.04
- 고용 형태: 정규직
- 핵심 포지션: 앱 배포와 독립된 모바일 웹뷰 제품
- 기술 스택: Next.js, React, TypeScript, Tailwind CSS, Phaser, Next Intl, Flutter InAppWebView, Web Audio API, Airbridge, Branch.io

## 핵심 요약

앱 안에서 이벤트·운세·게임·드로우를 빠르게 추가하고 수정할 수 있도록 Next.js 기반 모바일 웹뷰 제품을 초기 설계부터 개발했습니다. 기능을 단발성 이벤트 페이지로 쌓지 않고 도메인 단위로 나눠 앱 배포와 별개로 확장하게 했습니다.

## 1. 앱 배포와 독립된 제품 구조

- 이벤트, 운세, 게임, 기프트 코드, 드로우와 설문을 도메인 단위로 나눴습니다.
- 공통 앱바, 로더, 딥링크·공유 버튼과 다국어 리소스를 여러 콘텐츠에서 재사용하게 했습니다.
- Next.js 화면과 Phaser 게임 사이에 이벤트 브리지를 두어 게임 점수와 종료 상태를 화면으로 전달했습니다.

## 2. Flutter 앱과 웹뷰 연동

- `window.flutter_inappwebview.callHandler`로 공유, 알림, 화면 이동, 광고, 진동과 프로필 갱신 이벤트를 앱에 전달했습니다.
- Airbridge 딥링크와 친구 초대용 OG 메타데이터를 연결했습니다.
- 리워드 광고가 끝난 뒤 게임 이벤트와 화면 상태가 순서대로 이어지게 했습니다.

## 3. 게임 결과와 입력 상태 제어

- Phaser 게임의 점수 제출, 사용자 기록, 주간 랭킹과 보상 상태를 Next.js 화면에 연결했습니다.
- 결과 제출 중에는 게임 터치 입력을 막아 같은 결과가 중복 제출되지 않게 했습니다.
- 서버가 허용하지 않은 점수 응답에서는 성공 화면이나 다음 상태가 그대로 이어지지 않게 했습니다.

> 서버의 점수 검증 알고리즘을 개발한 것이 아니라, 프론트엔드에서 입력과 응답 이후 상태 전이를 통제한 범위입니다.

## 4. 모바일 브라우저와 생명주기 대응

- `env(safe-area-inset-bottom)`과 WebKit user-select·touch-callout 처리를 적용했습니다.
- Web Audio API에서 `webkitAudioContext` fallback을 사용하고, 사용자 상호작용 뒤에 오디오가 시작되도록 Safari 정책에 대응했습니다.
- `visibilitychange`에서 화면 이탈 시 오디오와 게임 상태를 멈추거나 초기화하고, 복귀 시 상태를 다시 맞췄습니다.
- Chrome, Safari와 Samsung Internet에서 키보드·포커스, 화면 안전 영역, 테마와 오디오 차이를 확인했습니다.

## 5. 결과

- 앱 배포와 별개로 게임과 이벤트를 추가·수정할 수 있는 제품 영역을 만들었습니다.
- 게임 배포 약 2주 뒤 한 주 동안 약 1,000명의 사용자가 게임을 이용했습니다.

> 주간 이용자 수는 단일 시점의 이용 규모입니다. 재방문율과 비교 기준이 없어 리텐션 상승으로 확대하지 않습니다.
