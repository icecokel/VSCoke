## 기본 정보

- 회사: 주식회사오프리메드
- 고용 형태: 정규직
- 직무: 서비스 개발자
- 직책: 개발팀 팀장
- 역할: 프론트엔드 개발, 일정·업무 배분, 팀원 면담, 프로젝트별 개발·검증 기준 관리

의료·임상 분석 제품의 프론트엔드를 개발하고 있습니다. 팀장으로 일정과 업무를 배분하고 팀원 면담을 진행하지만, 주 업무는 제품 개발이며 팀원이 막힌 구현이나 일정 위험이 큰 구간도 직접 맡아 해결합니다.

입사 당시 4개 프론트엔드 프로젝트는 프레임워크와 컨벤션, 런타임과 배포 방식이 서로 달랐고 배포는 FTP 중심이었습니다. 프로젝트를 옮길 때마다 실행 방법을 다시 파악해야 했던 문제를 줄이기 위해 실행·배포 스펙을 통일하고, CI/CD와 pre-push, Playwright E2E를 연결했습니다. AI 개발 보조 도구가 의료·임상 용어와 화면 상태를 임의로 해석하지 않도록 프로젝트별 지침·컨셉 문서와 Skill·Hook·PR·자동 검증 기준을 저장소에 연결했습니다.

## 1. 4개 프론트엔드 프로젝트의 실행·배포 스펙 통일

### 문제

- 프로젝트마다 프레임워크, Next.js·React 버전, package lock, Docker Node 런타임과 배포 스크립트가 달랐습니다.
- 저 역시 다른 프로젝트에 합류했을 때 실행과 배포 방법을 파악하는 데 시간이 필요했고, 같은 확인 과정이 팀원에게도 반복될 수 있었습니다.
- 패키지 버전만 맞춰서는 로컬 실행, 컨테이너 빌드와 실제 배포 결과가 달라지는 문제를 막기 어려웠습니다.

### 변경

- Vite·React 기반 프로젝트를 Next.js App Router와 standalone 실행 구조로 옮겼습니다.
- Next.js·React 패치 버전, package lock과 Docker Node 런타임을 프로젝트별로 확인해 같은 실행 기준으로 맞췄습니다.
- Docker 패키징과 load·deploy 스크립트가 동일한 빌드 산출물을 사용하도록 실행 경로를 통일했습니다.
- self-hosted runner 설치와 실행 순서를 문서화해 프로젝트별 설정을 다시 추적하지 않아도 되게 했습니다.

### 결과

- 4개 프로젝트에서 로컬 실행부터 컨테이너 패키징과 배포 전 확인까지 공통으로 사용할 기준을 남겼습니다.
- 팀원이 다른 프로젝트를 맡을 때 프레임워크와 배포 스크립트를 처음부터 추적하는 대신 같은 문서와 명령 순서로 확인할 수 있게 했습니다.

## 2. FTP 수동 배포를 CI/CD와 배포 전 검증으로 전환

### 문제

- FTP 배포는 매번 시간이 오래 걸렸고 배포 가능한 장소에도 제약이 있었습니다.
- typecheck와 build에서 발견할 수 있는 실패가 원격 브랜치나 배포 준비 단계까지 넘어갈 수 있었습니다.
- 브라우저와 화면 크기에 따른 회귀를 사람이 매번 같은 순서로 확인하기 어려웠습니다.

### 변경

- FTP 수동 배포를 GitHub Actions와 self-hosted runner 기반 CI/CD로 전환했습니다.
- 배포 서버에서 Docker 이미지를 빌드하고 컨테이너를 교체하도록 main·demo 배포 흐름을 연결했습니다.
- Husky pre-push에서 typecheck와 build를 실행해 실패한 변경이 원격 브랜치에 올라가지 않도록 막았습니다.
- Playwright E2E에 Chrome, Edge, Safari(WebKit)와 compact·wide 화면 크기를 넣어 핵심 사용자 흐름을 같은 명령으로 검증하게 했습니다.

### 결과

- 개발자가 있는 장소와 FTP 작업에 의존하던 배포를 GitHub Actions에서 실행할 수 있게 했습니다.
- 타입 오류와 빌드 실패는 push 전에 차단하고, 브라우저·화면 크기별 회귀는 반복 가능한 E2E 시나리오로 확인하게 했습니다.

## 3. AI 개발 보조 도구를 프로젝트 지침과 자동 검증 안에서 사용

### 문제

- 의료·임상 제품은 용어와 분석 흐름이 복잡해 AI가 화면 상태나 업무 맥락을 임의로 해석할 여지가 컸습니다.
- 프로젝트마다 구조와 규칙이 다른 상태에서 개인별 프롬프트만 사용하면 구현과 검토 기준이 달라질 수 있었습니다.
- AI가 생성한 코드도 기존 코드와 동일하게 typecheck, build와 브라우저 검증을 통과해야 했습니다.

### 변경

- 프로젝트별 도메인과 구현 원칙을 지침·컨셉 문서로 작성해 저장소에 배포했습니다.
- 반복 작업은 작업별 Skill로 분리하고, 실행 전후 확인 절차는 Hook에 연결했습니다.
- 코딩 컨벤션, 브랜치 전략과 PR 규칙을 문서화해 사람과 AI가 같은 기준을 참고하게 했습니다.
- 결과물은 Husky pre-push, typecheck, build와 Playwright 검증을 거치도록 기존 개발 흐름에 연결했습니다.

### 결과

- AI가 프로젝트 맥락을 추측하기보다 저장소에 남긴 지침과 작업 절차를 먼저 참고할 수 있게 했습니다.
- 개인 대화에 머물던 사용 방법을 팀원이 다시 사용할 수 있는 Skill, Hook과 PR 규칙으로 옮겼습니다.
- AI 활용 자체를 성과로 내세우기보다 결과를 검토하고 실패를 차단하는 기준을 함께 남겼습니다.

## 4. 이미지 중심 UI를 공통 컴포넌트와 빌드 시점 최적화로 전환

### 문제

- 일부 프레임과 아이콘이 정적 이미지에 의존해 불필요한 네트워크 요청이 발생했습니다.
- 버튼, 아이콘, 토글과 색상 사용 방식이 화면마다 달라 공통 UI를 수정할 때 여러 구현을 함께 확인해야 했습니다.
- 원본 이미지를 그대로 배포하면 실제 화면보다 큰 자산이 전송될 수 있었습니다.

### 변경

- 이미지 기반 glass frame을 CSS와 코드 기반 공통 컨테이너로 바꿨습니다.
- 버튼, 아이콘과 토글을 공통 컴포넌트로 옮기고 직접 색상 코드 529건을 디자인 토큰으로 전환했습니다.
- 정적 이미지 후보를 빌드 시 화면 크기별 WebP로 만들고, 입력과 정책이 같으면 기존 결과물을 재사용하게 했습니다.
- 이미지 파일·사용처·정책이 바뀐 경우에만 pre-push 검증을 실행하고, Docker 배포 전에 생성 결과물과 hash를 확인하게 했습니다.

### 결과

- 이미지 기반 프레임의 runtime PNG 요청 3건과 약 315 KiB payload를 제거했습니다.
- 이미지 후보의 전송량을 4.12 MB에서 136.0~228.1 KB 수준으로 낮춰 화면 크기와 DPR에 따라 약 94.6~96.8% 절감되는 것을 확인했습니다.
- Lighthouse 비교 기준 Performance는 88에서 98, LCP는 1.3초에서 1.0초, Speed Index는 3.4초에서 0.7초로 기록됐습니다.

> 이미지 전송량은 viewport와 DPR별 선택 후보를 계산한 값이며, Lighthouse 수치는 서로 다른 배포 환경의 비교 캡처입니다. 운영 사용자 전체를 반복 측정한 지표로 확대하지 않습니다.

## 5. 분석 작업별 데이터 격리와 재진입 복원

### 문제

- 같은 질병 목록을 여러 화면과 hook에서 직접 호출해 캐시 시간과 재요청 기준이 흩어져 있었습니다.
- 다른 분석 작업의 데이터가 섞이지 않아야 했고, 새로고침이나 재진입 후에도 사용자가 같은 조건과 결과를 확인할 수 있어야 했습니다.

### 변경·결과

- 화면과 hook에 남아 있던 질병 목록 직접 호출 6곳을 제거하고, 전용 TanStack Query hook에서 캐시와 재요청 기준을 관리하게 했습니다.
- 분석 데이터의 캐시 기준을 task identity에 맞추고, 결과와 적용 조건을 작업 단위로 다시 조회해 새로고침 후에도 같은 분석 상태를 복원하게 했습니다.
- 분석 결과에서 후속 task를 생성하고 basis, subgroup과 current step을 다음 화면에 전달해 결과 확인 이후의 분석 흐름이 이어지게 했습니다.
- 데이터 업로드를 청크 처리하고 직접 진입, 빈 작업과 조건 미선택처럼 잘못된 요청으로 이어지는 상태를 실행 전에 막았습니다.

## 6. 개발을 병행하는 팀장 역할

- 프론트엔드 개발을 주 업무로 수행하면서 일정과 업무를 배분하고 팀원 면담을 진행했습니다.
- 팀원이 막힌 구현이나 일정 위험이 큰 구간은 직접 개발해 다음 작업이 이어지도록 지원했습니다.
- 코드 리뷰는 설계 전반을 주도하는 방식보다 프로젝트 컨벤션과 구현 일관성을 확인하는 범위로 진행했습니다.
- 프로젝트별 실행 스펙, 배포·검증 절차와 AI 활용 지침을 문서로 공유해 반복되는 판단을 줄였습니다.
