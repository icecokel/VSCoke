## 1. 의료·임상 분석 제품 화면과 분석 데이터

### 문제

- 화면 이동과 새로고침 뒤에도 작업별 분석 결과와 조건을 같은 기준으로 복원해야 했습니다.
- 이미지 UI와 화면별 스타일이 흩어져 있어 수정 비용이 컸고, 정적 이미지 전송량도 줄일 필요가 있었습니다.

### 해결·결과

- 분석 데이터를 서버에 저장하고 작업별 캐시를 분리해, 재진입과 새로고침 후에도 같은 결과와 조건을 복원했습니다.
- 유효성 검사와 에러 케이스를 보완하고 데이터 업로드를 청크 처리했습니다.
- 이미지 UI를 코드 기반 컴포넌트로 바꾸고 버튼·아이콘·토글을 공통 구조로 전환했습니다. 직접 색상 코드 529건은 디자인 토큰으로 옮겼습니다.
- 정적 이미지를 빌드 시 화면 크기별 WebP로 만들고 변경분만 재생성·검증해 전송 용량을 약 95% 줄였습니다.
- Lighthouse 기준 LCP를 1.3초에서 1.0초로, Speed Index를 3.4초에서 0.7초로 낮췄습니다.

## 2. 팀 운영과 개발·검증 기준

### 문제

- FTP 수동 배포는 시간이 오래 걸리고 작업 장소가 제한됐습니다.
- 서로 다른 컨벤션과 프레임워크를 사용하던 4개 프론트엔드 프로젝트에 공통 실행·배포 기준이 필요했습니다.
- 팀원이 같은 기준으로 구현과 검토를 반복할 수 있는 자동 검증과 프로젝트 지침이 필요했습니다.

### 해결·결과

- 프론트엔드 개발과 함께 일정·업무 배분·코드 리뷰·면담을 맡고, 팀원이 막힌 구현과 일정 위험이 큰 구간을 직접 지원했습니다.
- FTP 수동 배포를 GitHub Actions CI/CD로 전환했습니다.
- 4개 프론트엔드 프로젝트의 실행·배포 스펙을 통일했습니다.
- Playwright E2E 테스트로 Chrome, Edge, Safari(WebKit)와 주요 화면 크기의 핵심 사용자 흐름을 자동 검증하고, Husky pre-push에서 typecheck·build 실패가 원격 브랜치에 올라가지 않도록 차단했습니다.
- 프로젝트별 지침·컨셉 문서를 팀에 배포하고, 저장소의 Skill·Hook과 PR 규칙에 연결했습니다.
