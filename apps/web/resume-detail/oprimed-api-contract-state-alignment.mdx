## 개요 (Overview)

**API 계약과 프론트엔드 상태 정합성 정리**는 Swagger/OpenAPI 계약 변경에 맞춰 요청·응답 타입, 서비스 레이어, 화면 상태 흐름을 정리한 작업입니다.

의료·임상 분석 화면은 입력 조건, 실행 조건, 결과 데이터가 서로 강하게 연결되어 있어 API 계약과 화면 상태가 어긋나면 런타임 오류나 잘못된 요청으로 이어질 수 있습니다. 이 작업에서는 mock과 임의 fallback 의존을 줄이고 실제 API payload 기준으로 화면 흐름을 맞췄습니다.

## 주요 구현 (Key Features)

### 1. API 타입과 서비스 레이어 정리

- **Swagger/OpenAPI 기준 정리**:
  - 환자, 질병, 약물, 시뮬레이션, 결과 API의 요청·응답 타입을 계약 기준으로 정리했습니다.
  - 화면에서 직접 payload를 조립하던 흐름을 서비스 레이어로 모아 요청 구조를 읽기 쉽게 만들었습니다.
- **mock/fallback 의존 축소**:
  - 임의 fallback과 mock 응답에 맞춰져 있던 화면 흐름을 실제 API payload 기준으로 보정했습니다.
  - 실행 조건, 입력값, 검증 조건이 API 계약과 맞도록 화면 상태를 재정리했습니다.

### 2. 입력 검증과 edge case 처리

- **도메인 입력 분기 처리**:
  - nullable 수치, 성별·인종 코드, 질병별 입력 분기, 약물 id, bootstrap 조건 등 edge case를 처리했습니다.
  - 잘못된 기본값이나 누락된 값이 시뮬레이션 요청으로 전송되지 않도록 보정했습니다.
- **실행/재진입 흐름 보정**:
  - 시뮬레이션 실행, 취소, 재진입, hydrate, import 복원 흐름에서 기존 값이 덮어써지거나 잘못 전송되는 문제를 정리했습니다.
  - API 응답 형태 변경에 맞춰 결과 화면이 안정적으로 렌더링되도록 보정했습니다.

### 3. 검증 근거 문서화

- **계약 변경 추적**:
  - API 계약 변경 내역과 프론트엔드 반영 범위를 문서화했습니다.
  - QA, design parity, progress 문서를 작성해 기능 구현 이후 검증 근거를 남겼습니다.

## 기술적 포인트 (Technical Depth)

- **TypeScript 타입 모델링**: API 요청·응답 타입을 명시하고, 화면 상태와 payload 변환 경계를 분리했습니다.
- **폼 검증**: `React Hook Form`, `Yup` 기반 검증 흐름에서 API 실행 조건과 화면 입력 조건을 맞췄습니다.
- **데이터 페칭**: `TanStack Query`를 통해 서버 데이터와 화면 상태의 책임을 분리했습니다.

## 결과 (Outcome)

API 계약, 화면 입력값, 실행 조건, 결과 렌더링 흐름을 맞춰 의료·임상 분석 화면의 런타임 오류 가능성과 잘못된 요청 가능성을 줄였습니다.
