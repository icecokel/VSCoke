## 핵심 요약

의료·임상 분석 제품의 프론트엔드 개발을 담당했습니다. 복잡한 분석 화면 구현뿐 아니라 분석 제품의 서버 연동 구조 전환, UI 컴포넌트 마이그레이션과 렌더링 성능 개선, 프론트엔드 실행 환경과 CI/CD 체계 구축을 맡았습니다.

## 1. 분석 제품의 서버 연동 구조 전환과 사용 흐름 구현

### 문제

- 같은 서버 데이터를 화면마다 직접 조회해 캐시, 재요청, 유지보수 기준이 흩어져 있었습니다.
- 분석 결과 이후 다음 분석으로 이어지는 화면 전환과 상태 전달 기준이 필요했습니다.
- 주요 사용 흐름 중 일부는 실제 동작 기준과 화면 상태 기준을 다시 맞춰야 했습니다.

### 해결·결과

- TanStack Query 기반 공통 조회 구조로 서버 데이터 조회 방식을 전환해, 질병 목록 직접 호출 6곳을 제거했습니다.
- 분석 결과 화면에서 task 생성, 상태 전달, 화면 전환 흐름을 구현해 후속 분석으로 이어지는 사용 흐름을 연결했습니다.
- 주요 사용 흐름에서 재정의가 필요한 구간과 동작하지 않는 구간을 분리해 공유했고, 에러·빈값·로딩 처리 기준 개선으로 이어지게 했습니다.

## 2. UI 컴포넌트 마이그레이션과 렌더링 성능 개선

### 문제

- 일부 UI가 이미지에 의존해 불필요한 네트워크 요청이 발생했습니다.
- 화면마다 다른 UI 구현 방식이 남아 있어 유지보수 비용이 커질 수 있었습니다.

### 해결·결과

- 이미지 기반 UI를 코드 기반 공통 컴포넌트로 마이그레이션해 runtime PNG 요청 3건과 약 315 KiB payload를 제거했습니다.
- Lighthouse 비교 기준 LCP를 1.3s에서 1.0s로, Speed Index를 3.4s에서 0.7s로 낮춰 렌더링 속도 개선을 확인했습니다.
- 정적 이미지 WebP 변환 흐름을 추가해 이미지 리소스 전송량을 약 95% 줄였습니다.
- 공통 컴포넌트 사용 기준을 도입해 legacy button import 25건과 정적 SVG 파일 34개를 제거했습니다.
- 직접 hex 색상 사용을 529건 줄여 색상 관리 기준을 개선했습니다.

## 3. 프론트엔드 실행 환경과 CI/CD 체계 구축

### 문제

- 프로젝트마다 의존성, 실행 환경, 배포 전 확인 기준이 달라 빌드와 검증 과정에서 같은 문제를 반복 확인해야 했습니다.
- 커밋·푸시 단계에서 잡을 수 있는 오류가 배포 준비 단계까지 넘어갈 수 있었습니다.

### 해결·결과

- 보안 패치, 의존성 잠금 파일, Docker 실행 환경, Node.js 버전을 맞춰 프론트엔드 실행 환경을 표준화했습니다.
- GitHub Actions 기반 CI/CD와 Husky pre-commit/pre-push hook을 도입해 커밋·푸시·배포 전 검증 흐름을 연결했습니다.
- Playwright CLI 기반 검증 체계를 도입해 3개 브라우저 단일 실행과 6개 브라우저·화면 크기 조합 matrix 실행이 통과하는 기준 사례를 만들었습니다.
