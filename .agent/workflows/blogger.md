## description: 블로그 포스트 생성 워크플로우

1. 사용자에게 다음 정보를 요청한다:
   - **제목 (Title)**
   - **카테고리** (`dev` 또는 `journal` 중 선택)
   - **설명 (Description)**
   - **필수 태그 (Tags)** (쉼표로 구분)
2. 입력받은 **제목**을 기반으로 영문 슬러그(slug)를 생성한다.
   - 예: "나의 첫 글" -> "my-first-post"
   - 사용자가 원한다면 직접 입력한 슬러그를 사용한다.
3. 오늘 날짜를 `YYYY-MM-DD` 형식으로 생성한다.
4. `src/posts/[카테고리]/[슬러그].mdx` 경로에 파일을 생성한다. 내용(Body)은 선택한 **카테고리**에 따라 다르게 작성한다.
   **공통 Frontmatter:**

   > **주의**: `title`은 URL에 영향을 줄 수 있으므로 슬러시(`/`) 문자는 사용하지 않는다.

   ```yaml
   ---
   title: "[제목]"
   date: "[오늘날짜]"
   description: "[설명]"
   tags: [[태그]] <!-- 필수 태그 포함 3개 이상의 태그를 작성 -->
   published: false
   ---
   ```

   **Case A: 카테고리가 `dev` (기술 깊게 파기) 일 때 본문:**

   > **작성 지침 (Tone & Manner)**:
   >
   > 1. **대상**: 이 기술을 처음 접하거나 더 깊게 알고 싶어 하는 동료 개발자.
   > 2. **어조**: 옆자리에서 친절하게 설명해 주듯이 부드럽고 전문적인 어조 (~해요/합니다).
   > 3. **깊이**: 단순한 기능 소개를 넘어, **"이게 왜 필요하지?", "어떻게 동작하지?"**라는 근원적인 물음에 답해야 한다.
   > 4. **필수**: 모든 코드 예시에는 **주석으로 라인별 상세 설명**을 달아 이해를 돕는다.
   > 5. **정보 출처**: 작성 전 반드시 **최신 정보**를 검색한다. 의견이 아닌 기술 전달 시에는 블로그보다 **공식 문서**를 최우선으로 참고하며, 공식 사이트 링크를 필수적으로 첨부한다.

   ```markdown
   # [제목]

   ## 1. 들어가며: 왜 이 기술에 주목해야 할까요?

   <!-- 독자의 흥미를 유발하고, 이 기술이 등장하게 된 배경이나 기존 방식의 문제점을 이야기하듯 설명 -->
   <!-- "여러분은 이런 경험 없으신가요?" 같은 공감 유도 -->

   ## 2. 핵심 원리와 특징 (Deep Dive)

   <!-- [Deep Dive] 단순 특징 나열이 아닌, 내부 동작 원리나 아키텍처를 그림을 그리듯 상세히 설명 -->
   <!-- 예: "Knip은 AST를 파싱하여..." 처럼 기술적인 메커니즘을 풀어서 설명 -->
   <!-- 공식 문서의 내용을 인용하되, 나만의 언어로 쉽게 해석 -->

   ## 3. 실무 적용 가이드 (With Code)

   <!-- "Hello World" 수준이 아닌, 실무에서 마주칠 법한 상황 예시 -->
   <!-- 주의: 코드 블록 내에 주석으로 라인별 상세 설명 추가 필수 -->

   ## 4. 주의사항 및 한계점

   <!-- 장점만 나열하지 말고, 사용 시 겪을 수 있는 트레이드오프나 주의할 점을 솔직하게 기재 -->

   ## 5. 마치며

   <!-- 전체 요약 및 이 기술을 통해 독자가 얻을 수 있는 이점 재강조 -->
   <!-- "이제 여러분의 프로젝트에도 적용해 보세요!" 같은 독려 -->
   ```

   **Case B: 카테고리가 `journal` (개발 일지) 일 때 본문:**

   ```markdown
   # [제목]

   ## 1. 고민과 문제점

   <!-- 작업을 시작하게 된 동기, 당시의 고민 -->

   ## 2. 선택지 분석

   <!-- 고민한 방법들과 각 방법의 장단점 -->

   ## 3. 해결 과정

   <!-- 해결 시도, 발생한 인상 깊은 사이드 이펙트 (생략 가능) -->

   ## 4. 최종 결과

   <!-- 최종 적용된 방법 -->

   ## 5. 후기

   <!-- 아직 걱정되는 부분이나 간단한 후기 -->
   ```

5. 작성된 본문 내용에서 가이드용 **주석(`<!-- ... -->`)은 모두 제거**한다.
6. 생성된 파일을 `view_file`로 확인시켜주고, 파일 생성 성공 메시지를 출력한다.
