# Playwright CLI 작업계획서

## 1. 문서 목적

이 문서는 `vscoke` 프로젝트에 대해 `Playwright CLI` 기반 E2E 테스트를 어떤 범위로 운영하고, 어떤 순서로 정비 및 확장할지 정의하는 작업계획서다.

참고 운영 규칙 문서:

- `docs/playwright-cli-test-spec.md`

## 2. 배경

현재 저장소에는 이미 Playwright 설정과 E2E 시나리오가 존재한다.

- 실행 설정: `apps/web/playwright.config.ts`
- 실행 래퍼: `apps/web/scripts/playwright-runner.mjs`
- 반복 실행 서버: `apps/web/scripts/playwright-web-server.mjs`
- 테스트 디렉토리: `apps/web/tests/e2e`

즉, 이번 작업은 Playwright를 처음 도입하는 단계라기보다, 이미 존재하는 E2E 체계를 `CLI 중심 표준 흐름`으로 정리하고 테스트 범위를 운영 가능한 형태로 확정하는 단계다.

## 3. 작업 목표

### 3-1. 1차 목표

- 로컬 기준 E2E 실행 흐름을 `pnpm e2e:*` 명령 체계로 고정한다.
- 핵심 사용자 흐름에 대한 회귀 범위를 문서화한다.
- 테스트 우선순위를 `P0`, `P1`, `P2`로 구분한다.
- 현재 커버되지 않거나 백엔드 의존이 큰 범위는 명시적으로 제외하거나 후속 과제로 분리한다.

### 3-2. 완료 상태 정의

아래 조건을 만족하면 1차 작업 완료로 본다.

1. 팀원이 문서만 보고 로컬에서 Playwright CLI 흐름을 실행할 수 있다.
2. 핵심 회귀 범위와 제외 범위가 문서상 명확하다.
3. 신규 기능 추가 시 어떤 E2E를 같이 수정해야 하는지 판단할 수 있다.
4. 테스트 실패 시 산출물과 확인 경로가 정리되어 있다.

## 4. 테스트 범위 정의

### 4-1. P0 범위

릴리즈 전 기본적으로 깨지면 안 되는 핵심 사용자 흐름이다.

| 범주           | 범위                                                       | 현재 상태    |
| -------------- | ---------------------------------------------------------- | ------------ |
| 로케일         | 루트 진입, 언어 전환, 쿠키 유지, 새로고침, 루트 리다이렉트 | 커버 중      |
| 홈             | Hero CTA, Quick Launch 카드 이동                           | 커버 중      |
| 네비게이션     | 메뉴바, 사이드바, 탐색기, 검색 패널                        | 커버 중      |
| README/Resume  | 프로필 진입, 상세 이력서 이동, 뒤로 이동, 공유 버튼        | 커버 중      |
| Blog           | 목록 진입, 상세 진입, 대시보드 검색                        | 커버 중      |
| Game Dashboard | Sky Drop, Fish Drift, Poke Lounge, Wordle 진입             | 커버 중      |
| Package        | 패키지 화면 렌더링                                         | 간접 커버 중 |
| 핵심 복구      | 404 복구, history 탭, 새로고침 상태 유지                   | 커버 중      |

P0 범위는 최소 `pnpm e2e`에서 안정적으로 검증되어야 한다.

### 4-2. P1 범위

기능 품질을 높이기 위한 보강 범위다.

| 범주      | 범위                                              | 현재 상태 |
| --------- | ------------------------------------------------- | --------- |
| 장애 대응 | 랭킹 API 실패, 단어 API 실패, capability fallback | 커버 중   |
| 접근성    | 키보드 전용 조작                                  | 커버 중   |
| 모바일    | 사이즈별 동작, 모바일 전용 트리거                 | 커버 중   |
| 딥링크    | 상세 라우트 직접 진입                             | 커버 중   |
| 시각 회귀 | 홈, 블로그, 대시보드, 게임, 패키지 화면           | 커버 중   |

P1은 기능 배포 직전 또는 UI 변경 시 반드시 확인하는 범위다.

### 4-3. P2 범위

현 시점에서 후속 정비가 필요한 범위다.

| 범주                 | 범위                                                 | 필요 작업                                                     |
| -------------------- | ---------------------------------------------------- | ------------------------------------------------------------- |
| 공유 결과 페이지     | `/{locale}/share/[id]` 진입 및 fallback              | 서버 사이드 API mocking 전략 필요                             |
| 실제 인증 연동       | 로그인 상태에 따른 점수 제출                         | 인증 세션 제어 전략 필요                                      |
| 교차 브라우저 정례화 | WebKit, Firefox 상시 회귀                            | 실행 시간과 flaky 관리 기준 필요                              |
| CI 확장              | 전체 E2E, visual regression, cross-browser 자동 회귀 | 기본 PR focused workflow는 구현됨. 확장 범위만 별도 결정 필요 |

P2는 1차 도입 완료 조건에 포함하지 않는다.

### 4-4. 명시적 제외 범위

이번 단계에서 아래 항목은 기본 회귀 범위에 포함하지 않는다.

- 실제 외부 API 서버 가용성 자체 검증
- 실제 OAuth 로그인 플로우
- Open Graph 이미지 픽셀 검증
- `robots.ts`, `sitemap.ts` 같은 SEO 산출물 검증
- Phaser 게임의 점수 공식, 물리 밸런스, 프레임 단위 판정 정확성

이 항목들은 성격상 E2E보다 별도 테스트 또는 수동 검증 전략이 더 적합하다.

## 5. 현재 커버리지 요약

현재 존재하는 E2E 파일 기준 커버리지는 아래와 같다.

| 파일                          | 책임 범위                                       |
| ----------------------------- | ----------------------------------------------- |
| `i18n-integrity.spec.ts`      | 로케일 전환 및 유지                             |
| `core-routes.spec.ts`         | 홈, 메뉴, 사이드바, README, Blog, Game 주요 CTA |
| `deep-link.spec.ts`           | 주요 상세 페이지 직접 진입                      |
| `error-fallback.spec.ts`      | API 오류 fallback                               |
| `capability-fallback.spec.ts` | 브라우저 capability fallback                    |
| `history-tabs.spec.ts`        | 탭 상태머신                                     |
| `hobby-games.spec.ts`         | 게임 센터와 주요 게임 라우트 진입               |
| `hobby-recipes.spec.ts`       | 취미 레시피 페이지                              |
| `hobby-espresso.spec.ts`      | 에스프레소 기록 페이지                          |
| `state-persistence.spec.ts`   | 로컬 상태 복원                                  |
| `not-found-recovery.spec.ts`  | 404 자동 복구                                   |
| `keyboard-only.spec.ts`       | 키보드 조작                                     |
| `mobile-behavior.spec.ts`     | 모바일 전용 동작                                |
| `visual-regression.spec.ts`   | 비주얼 기준선                                   |

즉, P0와 P1의 상당 부분은 이미 구현되어 있고, 이번 작업계획의 핵심은 이를 운영 기준으로 고정하고 남은 공백을 분리하는 것이다.

## 6. 단계별 작업 계획

### 6-1. Phase 1. 실행 규칙 고정

목표:

- Playwright CLI 공식 실행 경로를 문서화한다.
- 자동 서버 기동 흐름과 직접 CLI 흐름의 차이를 명확히 한다.

작업:

- `pnpm e2e:*` 명령 체계 정리
- 서버 자동 기동 규칙 정리
- 포트, worker, cross-browser 환경 변수 정리
- 스냅샷 갱신 규칙 정리

산출물:

- `docs/playwright-cli-test-spec.md`

### 6-2. Phase 2. 범위 기준 확정

목표:

- 어떤 사용자 흐름을 기본 회귀로 볼지 합의한다.
- 현재 커버 중인 항목과 미대상 항목을 분리한다.

작업:

- P0/P1/P2 범위 분류
- 제외 범위와 선행조건 명시
- 파일별 책임 범위 문서화

산출물:

- 본 문서 `docs/playwright-cli-test-plan.md`

### 6-3. Phase 3. 공백 보강

목표:

- 문서상 P0 또는 P1인데 아직 불안정한 항목을 보강한다.

우선 후보:

- `package` 화면의 기능성 assertion 보강 여부 확인
- 게임 대시보드 하위 라우트의 시작/복귀 흐름 안정화
- 모바일/키보드 시나리오 flaky 제거
- 공통 helper 추가 정리

완료 기준:

- 관련 파일 수정 후 `pnpm e2e`
- 필요 시 `pnpm e2e:update-snapshots`

### 6-4. Phase 4. 후속 확장

목표:

- 현재 제외된 범위 중 자동화 가치가 높은 항목을 순차 편입한다.

우선순위:

1. `share/[id]` 테스트 전략 확정
2. 크로스 브라우저 정례 실행 기준 확정
3. 전체 E2E, visual regression, cross-browser용 별도 CI job 검토

## 7. 테스트 실행 정책

### 7-1. 기본 회귀

아래 상황에서는 `pnpm e2e`를 기본 실행으로 본다.

- 메뉴바, 사이드바, 히스토리 탭 변경
- i18n 변경
- 블로그/이력서/공유 UI 변경
- 게임 대시보드 및 게임 진입 흐름 변경

### 7-2. 부분 실행

아래 상황에서는 단일 spec 실행을 허용한다.

- 국소 버그 수정
- 셀렉터 조정
- 특정 페이지 회귀 확인

예시:

```bash
pnpm e2e:server
PLAYWRIGHT_BASE_URL=http://127.0.0.1:37123 pnpm --filter @vscoke/web exec playwright test tests/e2e/history-tabs.spec.ts --project=chromium
```

### 7-3. 비주얼 실행

아래 상황에서는 비주얼 기준선 확인이 필요하다.

- 레이아웃 변경
- 색상, 간격, 타이포, 카드 배치 변경
- 홈, 블로그, 게임, 패키지 페이지 구조 변경

명령:

```bash
pnpm e2e:update-snapshots
```

## 8. 리스크와 대응

| 리스크        | 설명                                                                                     | 대응                             |
| ------------- | ---------------------------------------------------------------------------------------- | -------------------------------- |
| 게임 UI flaky | Phaser 로딩 시점과 상호작용 타이밍이 흔들릴 수 있음                                      | 명시적 ready 조건과 test id 사용 |
| 외부 API 의존 | 공유 결과 페이지처럼 SSR fetch가 필요한 구간은 브라우저 route mocking으로 부족할 수 있음 | 서버 레벨 stub 전략 분리         |
| 스냅샷 노이즈 | 플랫폼별 렌더 차이로 비주얼 diff가 흔들릴 수 있음                                        | Chromium 기준선만 유지           |
| 텍스트 변경   | 번역 문구 변경 시 locator가 흔들릴 수 있음                                               | 메시지 JSON 기반 locator 유지    |

## 9. 최종 수용 기준

이 계획서는 아래 판단에 활용할 수 있어야 한다.

- 무엇을 기본 회귀로 볼지 빠르게 결정할 수 있다.
- 어떤 테스트가 이미 있는지와 어떤 범위가 비어 있는지 한 번에 파악할 수 있다.
- 신규 기능 추가 시 Playwright 테스트를 어디까지 같이 수정해야 하는지 판단할 수 있다.
- 후속 과제를 바로 티켓으로 쪼갤 수 있다.
