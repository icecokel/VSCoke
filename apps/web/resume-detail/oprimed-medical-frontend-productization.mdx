## 기본 정보

- 회사: 주식회사오프리메드
- 고용 형태: 정규직
- 직무: 서비스 개발자
- 역할: 프론트엔드 개발, 제품 흐름·실행 제어, 실행 구조·배포 전환, 개발·검증 기준 관리

의료·임상 분석 제품의 프론트엔드를 개발하고 있습니다. 환자·질병·약물·분석 조건이 연결된 화면 순서와 API 계약을 기획·디자인·백엔드 담당자와 협의하고, 화면 이동·새로고침·재진입에도 같은 작업의 조건과 결과가 이어지도록 작업 데이터를 복원했습니다.

Vite·React Router 기반 Portal은 Next.js App Router와 standalone Node 구조로 옮겼고, Docker tar 생성·SFTP 전송·서버 load에 의존하던 Nexus FE 배포는 self-hosted runner 기반 GitHub Actions로 전환했습니다. 출시 전 변경 전파 누락은 로컬 Git Hook에서 원격 반영 전에 중단하고, Nexus TOP의 핵심 흐름은 별도 Playwright 시나리오로 검증했습니다.

## 1. Portal 실행 구조 전환

### 문제

- Portal은 Vite·React Router로 실행됐고, 다른 프론트엔드 프로젝트는 Next.js와 Docker 중심으로 실행·배포됐습니다.
- 패키지만 맞추는 것으로는 로컬 실행, 컨테이너 빌드와 실제 배포 결과의 차이를 줄이기 어려웠습니다.

### 변경

- Vite·React 기반 프로젝트를 Next.js App Router와 standalone 실행 구조로 옮겼습니다.
- 다른 프론트엔드 프로젝트와 패키지, Docker Node 런타임과 배포 스펙을 맞췄습니다.
- Portal의 기존 tar 배포 스크립트가 컨테이너 기동 여부뿐 아니라 `/healthz` 응답까지 확인하게 했습니다.

### 결과

- Portal을 다른 프론트엔드 프로젝트와 같은 Next.js·standalone Node 실행 경로에서 다룰 수 있게 했습니다.
- `/healthz` 확인 범위는 Portal의 기존 tar 배포 스크립트이며, 모든 GitHub Actions 배포의 서비스 검증으로 확대하지 않습니다.

## 2. Nexus FE 배포와 변경 전파 검증

### 문제

- Nexus FE 배포는 Docker tar 생성, SFTP 전송과 서버 load를 개발자가 직접 실행해야 했습니다.
- API 스키마·타입·파일명 변경이 함께 전파되지 않으면 typecheck·test·build에서 실패할 수 있었습니다.
- 핵심 사용자 흐름은 브라우저와 화면 크기별로 같은 순서로 다시 확인할 수 있어야 했습니다.

### 변경

- self-hosted runner에서 환경별 Docker 이미지를 Git SHA로 빌드·교체하는 GitHub Actions 배포로 전환했습니다.
- 새 컨테이너가 정해진 시간 안에 running 상태가 되지 않으면 직전 이미지를 다시 기동하게 했습니다.
- typecheck·test·build 오류는 `pre-commit`과 `pre-push`에서 원격 반영 전에 중단했습니다.
- 별도 Playwright 시나리오에서 Nexus TOP 핵심 흐름을 Chrome·Edge·WebKit과 두 데스크톱 화면 크기로 검증했습니다.

### 결과

- Docker tar와 SFTP 작업에 의존하던 Nexus FE 배포를 GitHub Actions에서 실행할 수 있게 했습니다.
- 직전 이미지 재기동은 새 컨테이너 기동 실패를 복구하는 범위이며, 환경변수·DB·API 계약까지 되돌리는 전체 시스템 rollback으로 설명하지 않습니다.
- Git Hook은 우회 가능한 로컬 기본 경로이고, Playwright는 별도 실행 시나리오입니다.

## 3. AI 개발 보조 도구를 프로젝트 지침과 자동 검증 안에서 사용

### 문제

- 의료·임상 제품은 용어와 분석 흐름이 복잡해 AI가 화면 상태나 업무 맥락을 임의로 해석할 여지가 컸습니다.
- 프로젝트마다 구조와 규칙이 다른 상태에서 개인별 프롬프트만 사용하면 구현과 검토 기준이 달라질 수 있었습니다.
- AI가 생성한 코드도 기존 코드와 동일하게 typecheck, build와 브라우저 검증을 통과해야 했습니다.

### 변경

- 프로젝트별 도메인과 구현 원칙을 콘셉트·에이전트 지침으로 저장소에 남겼습니다.
- commit·PR 기준을 문서화해 사람과 AI 개발 보조 도구가 같은 검토 기준을 참고하게 했습니다.
- 다른 팀원이 자신의 작업에 맞게 지침을 보완해 재사용했습니다.

### 결과

- AI가 프로젝트 맥락을 추측하기보다 저장소에 남긴 지침과 작업 절차를 먼저 참고할 수 있게 했습니다.
- 개인 대화에 머물던 사용 방법을 팀원이 보완하고 재사용할 수 있는 저장소 기준으로 옮겼습니다.
- AI 활용 자체보다 의료·임상 용어와 화면 동작의 임의 해석을 줄이고, 결과를 검토할 기준을 남겼습니다.

## 4. 이미지 중심 UI를 공통 컴포넌트와 빌드 시점 최적화로 전환

### 문제

- 일부 프레임과 아이콘이 정적 이미지에 의존해 불필요한 네트워크 요청이 발생했습니다.
- 버튼, 아이콘, 토글과 색상 사용 방식이 화면마다 달라 공통 UI를 수정할 때 여러 구현을 함께 확인해야 했습니다.
- 원본 이미지를 그대로 배포하면 실제 화면보다 큰 자산이 전송될 수 있었습니다.

### 변경

- 이미지 기반 glass frame을 CSS와 코드 기반 공통 컨테이너로 바꿨습니다.
- 버튼, 아이콘과 토글을 공통 컴포넌트로 옮기고 직접 색상 코드 529건을 디자인 토큰으로 전환했습니다.
- 정적 이미지 후보를 빌드 시 화면 크기별 WebP로 만들고, 입력과 정책이 같으면 기존 결과물을 재사용하게 했습니다.
- 이미지 파일·사용처·정책이 바뀐 경우에만 pre-push 검증을 실행하고, Docker 배포 전에 생성 결과물과 hash를 확인하게 했습니다.

### 결과

- 이미지 기반 프레임의 runtime PNG 요청 3건과 약 315 KiB payload를 제거했습니다.
- 이미지 후보의 전송량을 4.12 MB에서 136.0~228.1 KB 수준으로 낮춰 화면 크기와 DPR에 따라 약 94.6~96.8% 절감되는 것을 확인했습니다.
- Lighthouse 비교 기준 Performance는 88에서 98, LCP는 1.3초에서 1.0초, Speed Index는 3.4초에서 0.7초로 기록됐습니다.

> 이미지 전송량은 viewport와 DPR별 선택 후보를 계산한 값이며, Lighthouse 수치는 서로 다른 배포 환경의 비교 캡처입니다. 운영 사용자 전체를 반복 측정한 지표로 확대하지 않습니다.

## 5. 분석 작업별 데이터 격리와 재진입 복원

### 문제

- 같은 질병 목록을 여러 화면과 hook에서 직접 호출해 캐시 시간과 재요청 기준이 흩어져 있었습니다.
- 다른 분석 작업의 데이터가 섞이지 않아야 했고, 새로고침이나 재진입 후에도 사용자가 같은 조건과 결과를 확인할 수 있어야 했습니다.

### 변경·결과

- 화면과 hook에 남아 있던 질병 목록 직접 호출 6곳을 제거하고, 전용 TanStack Query hook에서 캐시와 재요청 기준을 관리하게 했습니다.
- 분석 데이터의 캐시 기준을 task identity에 맞추고, 결과와 적용 조건을 작업 단위로 다시 조회해 새로고침 후에도 같은 분석 상태를 복원하게 했습니다.
- 분석 결과에서 후속 task를 생성하고 basis, subgroup과 current step을 다음 화면에 전달해 결과 확인 이후의 분석 흐름이 이어지게 했습니다.
- 데이터 업로드를 청크 처리하고 직접 진입, 빈 작업과 조건 미선택처럼 잘못된 요청으로 이어지는 상태를 실행 전에 막았습니다.

## 6. 장시간 분석의 중단 경계

- 장시간 ML 분석에서 프론트엔드 요청 중단과 서버 실행 중단 API 호출을 하나의 취소 흐름으로 연결했습니다.
- 취소 뒤 도착한 지연 응답은 화면에 반영하지 않고, 실행 상태와 진행 UI를 정리했습니다.
- 중단 시점 이후의 응답이 화면 상태를 다시 바꾸지 못하도록 프론트엔드 상태 전이 경계를 두었습니다.

## 7. 약 6주간의 공식 팀장 역할

- 3명 프론트엔드 팀에서 약 6주간 공식 팀장을 맡아 일정과 업무를 배분하고 팀원 면담을 진행했습니다.
- 팀원이 막힌 구현이나 일정 위험이 큰 구간은 직접 개발해 다음 작업이 이어지도록 지원했습니다.
- 코드 리뷰와 위험 구간 지원을 맡고, 프로젝트별 실행 스펙·배포·검증 절차와 AI 활용 지침을 문서로 공유했습니다.
