## 핵심 요약

의료·임상 분석 제품의 입력 화면, 결과 화면, 차트와 분석 데이터 저장·조회 흐름을 다뤘습니다. 화면 구현뿐 아니라 분석 요청이 잘못 넘어가지 않게 막고, 같은 데이터 기준으로 다시 조회되게 하는 구조를 함께 설계했습니다.

프로젝트별로 달랐던 실행·배포 방식을 맞추고, 변경 사항을 올리기 전에 실패를 더 빨리 확인할 수 있게 했습니다. 주요 브라우저와 화면 크기를 확인하는 E2E 테스트와 AI 기반 개발 환경도 함께 다뤘습니다.

## 1. 의료·임상 분석 제품 화면과 분석 데이터

### 문제

- 분석 입력 화면, 결과 화면, 차트, 파일 업로드·다운로드처럼 제품 화면마다 다뤄야 하는 상태와 데이터가 많았습니다.
- 화면과 hook에 같은 서버 데이터를 직접 조회하는 코드가 남아 있어 조회 기준이 흩어질 수 있었습니다.
- 분석 작업 데이터는 화면 이동과 재진입 뒤에도 같은 기준으로 다시 조회되어야 했습니다.
- 분석 요청 전 유효성 검사, 에러 케이스, 대용량 데이터 업로드 처리가 흔들리면 잘못된 요청이 서버로 넘어갈 수 있었습니다.

### 해결·결과

- 분석 입력 화면, 결과 화면, 차트, 파일 업로드·다운로드와 같은 제품 화면을 개발했습니다.
- 화면과 hook에 남아 있던 질병 목록 직접 호출 6곳을 제거하고, 같은 조회 기준을 한 곳에서 관리했습니다.
- 분석 작업 데이터를 서버에 저장하는 구조와 클라이언트 캐시 전략을 설계해, 화면 이동과 재진입 시 같은 데이터 기준으로 조회되게 했습니다.
- 부족했던 유효성 검사와 에러 케이스를 보강하고, 데이터 업로드를 청크 단위로 나눠 처리했습니다.
- 이미지로 표현하던 UI를 공통 컴포넌트와 CSS로 전환해 불필요한 이미지 요청과 수정 비용을 줄였습니다.
- Lighthouse 기준 LCP를 1.3s에서 1.0s로, Speed Index를 3.4s에서 0.7s로 낮췄습니다.

## 2. 개발 환경과 검증 시스템

### 문제

- 4개 프론트엔드 프로젝트의 실행 방식과 배포 방식이 달라 같은 문제를 반복 확인해야 했습니다.
- 변경 사항을 올리기 전 타입 오류와 빌드 실패를 더 빨리 확인할 필요가 있었습니다.
- 주요 브라우저와 화면 크기에서 핵심 화면 흐름을 반복 확인할 수 있는 테스트 기준이 필요했습니다.
- 의료·임상 도메인에서는 AI 개발 보조 도구가 어려운 도메인 용어와 화면 상태를 임의로 해석하지 않도록 프로젝트 맥락을 남겨야 했습니다.

### 해결·결과

- 4개 프론트엔드 프로젝트의 실행 방식과 배포 방식을 통일했습니다.
- GitHub Actions 기반 배포와 검증 흐름을 정리했습니다.
- 변경 사항을 올리기 전에 타입 오류와 빌드 실패를 확인하게 해 실패한 변경이 원격 브랜치에 올라가지 않게 했습니다.
- 주요 브라우저와 화면 크기에서 핵심 화면 흐름을 확인하는 테스트 시스템을 만들었습니다.
- 어려운 도메인 용어와 화면 상태를 임의로 해석하지 않도록 작업별 스킬, 실행 훅, LLM Wiki, PR 확인 방식으로 프로젝트 맥락을 남겼습니다.
